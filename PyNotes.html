Python Fundamentals

Syntax & Style

  • Indentation matters. Python uses whitespace to define code blocks — unlike JS or C++. Mixing tabs and spaces will cause errors.
  • Naming convention: use snake_case for variables and functions (e.g. player_health, calculate_damage).
  • Comments:
    # Single-line comment

"""
Multi-line comment (technically a string literal,
but commonly used as a docstring or block comment)
"""

Variables & Types

Variables are declared without a keyword — just assign directly:

sword_damage = 10
player_name = "Arthas"
is_alive = True
empty_slot = None   # NoneType — represents an intentionally empty/absent value

Common types:

Type Example
int 42
float 3.14
str "hello"
bool True / False
NoneType None

Check a variable's type with type():

type(sword_damage)   # <class 'int'>

Strings & Printing

f-strings are the cleanest way to embed values in strings:

stat = 42
print(f"The value of stat is: {stat}")

Concatenation with + (types must match — can't mix str and int directly):

print("Player Health: " + str(player_health) + " HP")

Math & Operators

Operator Description Example Result
+ - * / Standard arithmetic 10 / 3 3.333...
// Floor division (rounds down) 7 // 3 2
% Modulo (remainder) 7 % 3 1
** Exponent 3 ** 2 9

Assignment shorthands (++/-- do NOT exist in Python):

health += 10    # add
health -= 5     # subtract
health *= 2     # multiply
health /= 2     # divide

Readability tricks for big numbers:

big_number = 10_000_000      # underscore as thousands separator
sci_notation = 1.024e18      # scientific notation

Binary / Bitwise:

# Prefix binary literals with 0b
can_delete_mask = 0b0001

0b0001 & 0b1001   # AND  → 0001 (both bits must be 1)
0b0001 | 0b1001   # OR   → 1001 (either bit can be 1)

# Convert binary string to int (base 2)
int(binary_variable, 2)

Logic & Comparison

Comparison operators: > < >= <= == !=

Logical operators (Python uses words, not symbols):

if health > 0 and mana > 0:   # both true
if health > 0 or mana > 0:    # at least one true
if not is_dead:                # negation

Contrast with JS/C where you'd write &&, ||, !

Control Flow

if health <= 0:
    return "dead"
elif health <= 5:
    return "injured"
else:
    return "healthy"

Loops

for loop with range():

for i in range(0, 100):       # 0 up to (not including) 100
    print(i)

for i in range(10, 0, -1):    # countdown — third arg is step
    print(i)

for i in range(0, 10, 2):     # step by 2
    print(i)

while loop:

while current_health < max_health and enemy_distance > 3:
    current_health += 1
    enemy_distance -= 2

Loop control:

continue   # skip to next iteration
break      # exit the loop entirely

Functions

Defined with the def keyword. Must be defined before they are called — unlike JS/C++, Python does not hoist functions.

def area_of_circle(radius):
    return 3.14 * radius * radius

Multiple return values (returns a tuple):

def get_stats():
    return damage, mana        # unpack with: dmg, mp = get_stats()

Default parameter values:

def take_damage(health, armor=0):   # armor defaults to 0 if not passed
    return health - max(0, damage - armor)

Scope

Variables defined outside a function are in global scope and accessible inside functions — but this can get messy. Best practice: pass values in explicitly. Globals are fine for true constants.

PLAYER_LEVEL = 4   # constant — ALL_CAPS is the convention for constants

def calculate_health(modifier):
    return PLAYER_LEVEL * modifier  # readable here, but prefer passing it in

Lists

Ordered, mutable (changeable) collections. Indexed from 0.

inv_list = ["Healing Potion", "Leather Scraps", "Iron Helmet", "Shortsword"]

inv_list[0]       # "Healing Potion"
len(inv_list)     # 4

inv_list.append("Shield")   # add to end
inv_list.pop()               # remove & return last item
inv_list.pop(0)              # remove & return item at index 0

del inv_list[0]              # delete by index (no return)
del inv_list[-2:]            # delete a slice

Iterating without indexes:

trees = ['oak', 'pine', 'maple']
for tree in trees:
    print(tree)

Slicing — list[start:end:step]

  • Negative indexes count from the end: -1 = last item, -2 = second to last
champions[2:]       # from index 2 to end
champions[:-1]      # from start up to (not including) last item
champions[0::2]     # every other item starting at index 0

Combining lists:

fav_items = fav_weapons + fav_armor

Membership check:

return weapon in top_weapons    # True or False

Tuples

Ordered and immutable (can't be changed after creation). Use parentheses instead of brackets.

heroes = [
    ("Glorfindel", 2093, True),
    ("Gandalf",    1054, False),
    ("Gimli",       389, False),
]

Good for grouping related values that shouldn't change — like coordinates, RGB colors, or records in a list.

Strings — Useful Methods

.join() — glue a list of strings together with a delimiter:

words = ["hello", "there", "sam"]
sentence = " ".join(words)
# "hello there sam"

.split() — break a string into a list using a delimiter (defaults to any whitespace):

message = "hello there sam"
words = message.split()
# ["hello", "there", "sam"]

csv_line = "a,b,c"
parts = csv_line.split(",")
# ["a", "b", "c"]

Dictionaries

Store data as key: value pairs. Keys must be unique.

car = {
    "brand": "Toyota",
    "model": "Camry",
    "year": 2019,
}

# Add or update a key
player["coins"] = 0
player["coins"] = 1

# Delete a key
del names_dict["joe"]

# Check membership
if "joe" in names_dict: ...

Nested dictionaries:

progress["quests"]["bridge_run"]["status"]   # chain keys to drill down

Common pattern — counting with a dict:

def count_enemies(enemy_names):
    enemies_dict = {}
    for enemy_name in enemy_names:
        if enemy_name in enemies_dict:
            enemies_dict[enemy_name] += 1
        else:
            enemies_dict[enemy_name] = 1
    return enemies_dict

Sets

Unordered, no duplicates. Great for deduplication and membership checks.

spell_set = set()                          # empty set (can't use {} — that's a dict)
party_spell_set = set(party_spell_list)    # convert list → set, drops duplicates

spell_set.add("Fireball")
spell_set.remove("Fireball")

unique_to_set1 = set1 - set2              # set subtraction

Errors & Exceptions

Raising an error:

raise Exception("player id not found")

Catching errors:

try:
    print(get_player_record(1))
except Exception as e:
    print(e)

Catching specific error types (more specific first, general last):

try:
    10 / 0
except ZeroDivisionError:
    print("can't divide by zero")
except Exception as e:
    print(e)

Think of except blocks like elif — Python checks them top to bottom and runs the first match. Always put specific exceptions before the generic Exception catch-all.

Object-Oriented Programming

Actively being filled in — currently working through boot.dev's OOP in Python course.

Core idea: organize code around objects — bundles of data (attributes) and behavior (methods) — rather than just loose functions. An object is an instance of a class. The class is the blueprint; the object is the thing you actually use.

Key Pillars of OOP

  1. Encapsulation — bundling data + behavior together, hiding internal details
  2. Abstraction — exposing only what's necessary, hiding complexity
  3. Inheritance — child classes inherit and extend parent classes
  4. Polymorphism — different objects can share an interface but behave differently

These will be filled out as the course progresses.

Defining a Class

class Soldier:
    health = 5
    armor = 3
    damage = 2

Methods & self

Methods are functions defined inside a class. The first parameter is always self — Python uses it to refer to the specific object the method is being called on.

class Wall:
    armor = 10
    height = 5

    def fortify(self):
        self.armor *= 2         # modifies THIS wall's armor

    def get_cost(self):
        return self.armor * self.height    # getter pattern

Calling a method on an object:

soldier112.take_damage(damage, multiplier)

Constructors — __init__

__init__ runs automatically when you create a new object. It's how you set each instance's starting values. Methods with double underscores on each side are called dunder methods (short for "double underscore") — more on these later.

class Soldier:
    def __init__(self, name, armor, num_weapons):
        self.name = name
        self.armor = armor
        self.num_weapons = num_weapons

soldier_one = Soldier("Legolas", 2, 10)

You can also compute values in __init__:

class Wall:
    def __init__(self, depth, height, width):
        self.depth = depth
        self.height = height
        self.width = width
        self.volume = depth * height * width    # derived attribute

Objects Interacting with Other Objects

Methods can receive other object instances as parameters — not just primitives like ints or strings. This is how objects collaborate with each other.

class Archer:
    def __init__(self, name, health, num_arrows):
        self.name = name
        self.health = health
        self.num_arrows = num_arrows

    def take_hit(self):
        self.health -= 1
        if self.health <= 0:
            raise Exception(f"{self.name} is dead")

    def shoot(self, target):           # target is another Archer instance
        if not self.num_arrows:        # 0 is falsy — triggers when arrows run out
            raise Exception(f"{self.name} can't shoot")
        self.num_arrows -= 1
        print(f"{self.name} shoots {target.name}")
        target.take_hit()              # call a method on the OTHER object

Patterns worth noting:

  • Truthiness check: if not self.num_arrows is Pythonic shorthand for if self.num_arrows == 0. In Python, 0, None, "", and [] are all falsy — you can use them directly in conditionals without explicit comparison.
  • <= 0 vs == 0: When checking if something is depleted, prefer <= 0. If a value somehow goes negative (e.g. from a large hit skipping past 0), == 0 would never trigger. <= 0 is the defensive, correct choice.
  • Dot notation on the parameter: Because target is a full object, you have access to all its attributes and methods — target.name, target.health, target.take_hit(), etc. Same dot notation, different object.

Object Composition (has-a vs. is-a)

Objects can contain other objects as attributes. This is called composition — distinct from inheritance.

  • is-a → inheritance (a Dragon is a Monster)
  • has-a → composition (a Library has Books)
class Book:
    def __init__(self, title, author):
        self.title = title
        self.author = author

class Library:
    def __init__(self, name):
        self.name = name
        self.books = []          # Library "has" Books — stored in a list

    def add_book(self, book):
        self.books.append(book)

    def remove_book(self, book):
        # Build a new list excluding the match — safer than removing during iteration
        # (mutating a list while looping over it causes skipped elements)
        new_books = []
        for b in self.books:
            if b.title == book.title and b.author == book.author:
                continue
            new_books.append(b)
        self.books = new_books

    def search_books(self, search_string):
        matches = []
        for b in self.books:
            # .lower() on both sides = case-insensitive comparison
            if search_string.lower() in b.title.lower() or search_string.lower() in b.author.lower():
                matches.append(b)
        return matches

List comprehension alternative — a more idiomatic Python way to filter a list into a new one:

# These two are equivalent:

# Explicit loop version:
new_books = []
for b in self.books:
    if not (b.title == book.title and b.author == book.author):
        new_books.append(b)

# List comprehension version (one line):
self.books = [b for b in self.books if not (b.title == book.title and b.author == book.author)]

Read it as: "give me each b from self.books, but only if [condition]." Both are valid — the loop is more readable when starting out; the comprehension is more idiomatic once comfortable.

Public vs. Private Attributes & Methods

By default, all class attributes and methods are public — accessible from anywhere. Prefix with __ (double underscore) to make them private, restricting access to inside the class only.

class Wall:
    def __init__(self, armor, magic_resistance):
        self.__armor = armor                    # private
        self.__magic_resistance = magic_resistance  # private

    def get_defense(self):                      # public method exposing private data
        return self.__armor + self.__magic_resistance

front_wall = Wall(10, 20)
front_wall.__armor        # ❌ AttributeError — can't access from outside
front_wall.get_defense()  # ✅ 30

Private attributes are still accessible inside the class via self.__attr — just not from outside. This is encapsulation in practice: hide the internals, expose a controlled interface.

class Wizard:
    def __init__(self, name, stamina, intelligence):
        self.name = name                          # public
        self.__stamina = stamina                  # private
        self.__intelligence = intelligence        # private
        self.health = 100 * self.__stamina        # public, derived from private
        self.mana = 10 * self.__intelligence      # public, derived from private

Note: prefer referencing self.__attr over the raw constructor parameter when computing derived values. If the setter ever transforms the value before storing it, self.__attr picks up that transformation — the raw parameter would bypass it.

When to use private? Any internal data that outside code shouldn't directly read or modify. Expose it through methods (getters/setters) if access is needed — that way you control how it's used.

Exception Types — Using the Right One

Python has a hierarchy of built-in exception types. Using the right one makes code more precise and useful to callers.

raise ValueError("cannot deposit zero or negative funds")  # bad input value
raise TypeError("expected an int")                         # wrong type
raise Exception("something went wrong")                    # generic fallback

ValueError = the value passed in is inappropriate for the operation. Prefer it over the generic Exception when input validation fails — callers can catch it specifically if needed.

Input Validation Ordering

Validate inputs before checking business logic. This keeps the happy path clean and easy to follow:

# ❌ Fragile — business logic check wraps the input check
def withdraw(self, amount):
    if self.__balance >= amount:
        if amount <= 0:
            raise ValueError("cannot withdraw zero or negative funds")
        self.__balance -= amount
    else:
        raise ValueError("insufficient funds")

# ✅ Clear — guards run top to bottom, action only runs if all pass
def withdraw(self, amount):
    if amount <= 0:
        raise ValueError("cannot withdraw zero or negative funds")
    if self.__balance < amount:
        raise ValueError("insufficient funds")
    self.__balance -= amount

max() and min() as Guards

A clean way to clamp a value so it never goes below (or above) a threshold:

self.health -= max(0, fireball_damage - self.__stamina)
# ensures damage is never negative — high stamina resists fully rather than healing

Similarly, min() can cap a value from above:

self.health = min(self.health + heal_amount, self.max_health)
# ensures healing never exceeds the health cap

Abstraction with Private Helper Methods

Private methods (prefixed __) aren't just for data — they're useful for hiding internal logic too. The public interface stays clean; the implementation details stay tucked away.

def sprint_right(self):
    if self.__stamina <= 0:
        self.__raise_if_cannot_sprint()   # private helper
    self.__use_sprint_stamina()           # private helper
    self.move_right()
    self.move_right()

def __raise_if_cannot_sprint(self):
    raise Exception("not enough stamina to sprint")

def __use_sprint_stamina(self):
    self.__stamina -= 1

No else After a raise

Raising an exception immediately stops execution — so any code after an if/raise block only runs if the exception wasn't raised. The else is unnecessary:

# These are equivalent:
if self.__stamina <= 0:
    raise Exception("not enough stamina")
self.__use_sprint_stamina()   # only runs if no exception was raised

# vs. unnecessarily verbose:
if self.__stamina <= 0:
    raise Exception("not enough stamina")
else:
    self.__use_sprint_stamina()

Same principle applies after return — anything after a return in the same scope is unreachable.

DRY — Don't Repeat Yourself

If you're writing nearly identical code multiple times, that's a signal to refactor. One approach: pass a function as a parameter to a shared private helper.

# Instead of four nearly-identical sprint methods:
def __sprint(self, move_fn):
    if self.__stamina <= 0:
        raise Exception("not enough stamina to sprint")
    self.__stamina -= 1
    move_fn()
    move_fn()

def sprint_right(self):
    self.__sprint(self.move_right)

def sprint_left(self):
    self.__sprint(self.move_left)

Functions can be passed as arguments in Python — self.move_right without parentheses is a reference to the function itself (not a call to it).

__str__ — Dunder Method for Printing

__str__ defines what a print() call on an object displays. Another dunder method (like __init__), it's Python's way of letting you customize built-in behavior for your own classes.

class DeckOfCards:
    def __str__(self):
        return f"The deck has {len(self.__cards)} cards"

deck = DeckOfCards()
print(deck)   # "The deck has 52 cards"

Without __str__, print(deck) would show something like <__main__.DeckOfCards object at 0x...> — the default, useless representation.

__repr__ — Dunder Method for Developers

__repr__ is __str__'s sibling. While __str__ is the human-friendly display, __repr__ is meant to be an unambiguous, developer-facing representation — ideally one that could recreate the object.

class Rectangle:
    def __repr__(self):
        return f"Rectangle({self.__x1}, {self.__y1}, {self.__x2}, {self.__y2})"

r = Rectangle(1, 2, 3, 4)
repr(r)   # "Rectangle(1, 2, 3, 4)"

Quick rule of thumb: __str__ = "how should this look to a user?", __repr__ = "how would a developer recreate this?"

Nested Loops for Combinations

To generate every combination of two lists, nest one loop inside the other:

for suit in self.SUITS:
    for rank in self.RANKS:
        self.__cards.append((rank, suit))
# Produces all 52 (rank, suit) tuples in suit-major order

random.shuffle() — In-Place Shuffling

random.shuffle() modifies the list directly — it does not return a new list.

import random
random.shuffle(self.__cards)   # ✅ shuffles in place
self.__cards = random.shuffle(self.__cards)   # ❌ assigns None (common mistake)

Returning None Explicitly vs. Implicitly

A function that hits return None and a function that falls off the end without returning both produce None — but being explicit is clearer when None is a meaningful signal to the caller (e.g. "deck is empty"):

def deal_card(self):
    if not self.__cards:   # empty list is falsy
        return None
    return self.__cards.pop()

Inheritance

A child class can inherit from a parent class, gaining access to its public methods and attributes while adding its own.

class Human:
    def __init__(self, name):
        self.__name = name       # private to Human

    def get_name(self):
        return self.__name


class Archer(Human):             # Archer inherits from Human
    def __init__(self, name, num_arrows):
        super().__init__(name)   # call parent constructor first
        self.__num_arrows = num_arrows

    def get_num_arrows(self):
        return self.__num_arrows

Key syntax:

  • class Child(Parent): — declare inheritance in the class definition
  • super().__init__(...) — calls the parent's constructor; lets the parent set up its own private state

Private attributes don't cross class boundaries. Even though Archer inherits from Human, it cannot access Human's __name directly — private means private to that class. super().__init__() is how you let the parent handle its own internals.

self — When It Appears and When It Doesn't

self belongs in method definitions, never in method calls. Python passes the object reference automatically at call time.

def use_arrows(self, num):   # definition — self is explicit
    self.__num_arrows -= num

self.use_arrows(3)           # call — no self, just the actual arguments
target.get_name()            # call on another object — same rule

Under the hood, self.use_arrows(3) is Python doing Archer.use_arrows(self, 3) for you.

Inheritance Chains

Inheritance can chain multiple levels deep. A grandchild class inherits from the full chain:

class Human:      ...         # get_name()
class Archer(Human):   ...    # use_arrows(), get_num_arrows()
class Crossbowman(Archer): ...# triple_shot()

bolt = Crossbowman("William", 30)
bolt.get_name()      # ✅ inherited from Human
bolt.use_arrows(3)   # ✅ inherited from Archer
bolt.triple_shot(target)  # ✅ defined on Crossbowman

Every Crossbowman is also an Archer and also a Human.

Polymorphism — Method Overriding

Core idea: different classes can share the same method name but behave differently. Code that calls the method doesn't need to know which specific class it's dealing with — it just works.

class Unit:
    def in_area(self, x1, y1, x2, y2):
        # simple point check — is the unit's center inside the box?
        return (
            self.pos_x >= x1 and self.pos_x <= x2
            and self.pos_y >= y1 and self.pos_y <= y2
        )

class Dragon(Unit):
    def in_area(self, x1, y1, x2, y2):
        # override — uses full rectangle hit box instead of a single point
        area = Rectangle(x1, y1, x2, y2)
        return area.overlaps(self.__hit_box)

The Dragon class overrides the parent's in_area with its own version. Any code calling some_unit.in_area(...) works the same whether it's a basic Unit or a Dragon — that's polymorphism in action.

Operator Overloading — __add__

Dunder methods let you define how operators behave on your custom classes. __add__ controls the + operator.

class Sword:
    def __init__(self, sword_type):
        self.sword_type = sword_type

    def __add__(self, other):
        if self.sword_type == "bronze" and other.sword_type == "bronze":
            return Sword("iron")
        if self.sword_type == "iron" and other.sword_type == "iron":
            return Sword("steel")
        raise Exception("cannot craft")

iron_sword = Sword("bronze") + Sword("bronze")   # Sword("iron")

Under the hood, a + b calls a.__add__(b) — the left object's __add__ runs, with the right object passed as other.

Order matters. 5 + pouch calls int.__add__(5, pouch) — and int has no idea what a Pouch is, so it fails. To support swapped order, implement __radd__:

def __radd__(self, other):
    return self.__add__(other)    # Python tries this if the left side's __add__ fails

Full operator overloading reference:

Operation Operator Dunder Method
Addition + __add__
Subtraction - __sub__
Multiplication * __mul__
Power ** __pow__
Division / __truediv__
Floor Division // __floordiv__
Remainder (modulo) % __mod__
Bitwise Left Shift << __lshift__
Bitwise Right Shift >> __rshift__
Bitwise AND & __and__
Bitwise OR \| __or__
Bitwise XOR ^ __xor__
Bitwise NOT ~ __invert__

Same pattern for all of them — define the dunder method on your class, and the corresponding operator just works on your objects.

isinstance() — Type Checking

When an operator needs to handle different types coming in, isinstance() checks what kind of object you're dealing with:

def __add__(self, other):
    if isinstance(other, int):
        return Pouch(self.gold + other)       # Pouch + int
    if isinstance(other, Pouch):
        return Pouch(self.gold + other.gold)  # Pouch + Pouch
    raise TypeError("unsupported type")

Comparison Operator Overloading

Same pattern as arithmetic operators — define the dunder method, and the comparison operator works on your objects:

Operation Operator Dunder Method
Equal == __eq__
Not Equal != __ne__
Less Than < __lt__
Greater Than > __gt__
Less or Equal <= __le__
Greater or Equal >= __ge__ 
class Card:
    def __init__(self, rank, suit):
        self.rank = rank
        self.suit = suit
        self.rank_index = RANKS.index(rank)    # .index() returns position in a list
        self.suit_index = SUITS.index(suit)

    def __eq__(self, other):
        return self.rank_index == other.rank_index and self.suit_index == other.suit_index

    def __lt__(self, other):
        return self.rank_index < other.rank_index or (
            self.rank_index == other.rank_index and
            self.suit_index < other.suit_index
        )

.index() on lists — returns the position of an element. Handy for turning string values into comparable numbers:

["Clubs", "Diamonds", "Hearts", "Spades"].index("Hearts")   # 2

NotImplementedError — Enforcing Overrides

A parent class can use NotImplementedError to say "subclasses must override this method." If they forget, they get an explicit error instead of silent wrong behavior:

class Round:
    def resolve_round(self):
        raise NotImplementedError("Subclasses must implement resolve_round()")

class HighCardRound(Round):
    def resolve_round(self):       # override — this is required
        ...

Booleans Are Integers

In Python, True is 1 and False is 0 in arithmetic contexts. You can use this to collapse if/elif/else chains into a single expression:

# These are equivalent:
# if/elif version:
if card1 > card2:
    return 1
elif card1 < card2:
    return 2
return 0

# bool-as-int version:
return (card1 > card2) + 2 * (card1 < card2)

It's compact and clever, but the if/elif version is usually preferred in production code because the tie case (0) is explicit rather than implied by the math. Good trick to know, but readability wins in a team setting.
